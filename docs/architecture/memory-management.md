# 디버그 통신 및 메모리 관리 ⭐️

DSM은 다중 탭 환경에서 완벽하게 동작하는 `BroadcastChannel` 기반의 무설정 디버그 팝업을 제공합니다. 하지만 이 편리함 이면에는 치밀한 메모리 관리 전략이 숨어 있습니다.

## 1. 전역 변수 오용과 메모리 누수 방어

사용을 완료한 `BroadcastChannel`은 반드시 `close()`를 호출해야 합니다. 그렇지 않으면 GC 대상이 되지 못해 탭이 열려있는 동안 메모리 누수로 이어집니다.

## 2. Heartbeat (PING/PONG) 메커니즘

모바일 브라우저나 SPA 환경에서는 `beforeunload` 이벤트가 무시되거나 OOM(Out of Memory)으로 프로세스가 강제 종료될 위험이 큽니다.

* DSM은 이에 의존하지 않고, 디버거 팝업 내부에서 주기적으로 **`PING` 메시지(Heartbeat)**를 발송합니다.
* 각 탭은 자신이 살아있음을 증명하는 `PONG`으로 응답하며, 응답이 없는 탭의 데이터는 레지스트리에서 즉각 폐기(Eviction)되어 브라우저 힙 메모리를 안정적으로 유지합니다.

## 3. 명시적 객체 소멸자 제공

프로덕션 빌드에서 채널을 안전하게 닫고 완벽한 생명주기 관리를 지원하기 위해, 라이브러리 차원에서 `destroy()` 메서드를 제공하여 강제 GC 대상을 지정할 수 있게 설계되었습니다.
